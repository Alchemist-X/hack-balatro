# hack-balatro Backlog & 反思（2026-03-31）

## 诚实评估：当前到底什么水平

### 真实完成的

1. **基础架构**：Rust workspace 三 crate + Python binding + Gym wrapper，编译和测试全绿。
2. **核心游戏流程**：Blind 线性推进、Shop、Cashout、Ante 递增、Game Over 全链路可跑通。
3. **Consumable 框架**：买/卖/使用的管道已通，Planet 升级手牌类型 OK，11 张 Tarot 有真实效果。
4. **Retrigger 基础设施**：per-card 计分循环、Red Seal、Sock and Buskin、Hanging Chad、Seltzer、Dusk。Blueprint/Brainstorm 链式解析带环检测。
5. **Joker 覆盖面**：150 个 Joker 在 `apply_joker_effect` 都有匹配分支，不会再触发 `joker_not_implemented` 审计警告。
6. **三个新激活阶段**：held-in-hand / end-of-round / boss-blind-pre-play 有独立函数。
7. **审计通过**：zero error / zero warning / determinism 0 mismatch。

### 需要诚实承认的缺陷

#### 缺陷 1：28 个 "假 supported" Joker

28 个 Joker 标记了 `supported = true` 但实际效果是 **空的或写死初始值**，靠 TODO 注释糊弄过去了。审计不检查效果数值正确性，只检查 `supported` 标志位，所以 zero-warning 是"结构性通过"，不是"语义正确"。

这些 Joker 包括所有 **scaling Joker**（Ride the Bus、Green Joker、Constellation、Vampire、Lucky Cat、Campfire、Madness 等）和一些 **tracking Joker**（Red Card、Flash Card、Fortune Teller、Supernova 等）。它们都需要在 `JokerInstance` 上维护运行时状态（每次打牌/弃牌/跳盲/卖卡后更新），目前完全缺失。

**影响**：在有这些 Joker 的局面下，engine 的得分/经济计算是错的。对 agent 训练来说是有毒的训练信号。

#### 缺陷 2：审计只检查结构，不检查数值

当前 `audit_replay.py` 的检查全部是：
- trace 结构是否完整（joker_resolution 数量是否匹配）
- 状态转移是否合法（stage 切换、Lua state 对应）
- RNG trace 是否出现

**它完全不检查**：
- chips/mult 计算是否正确
- 经济数值（money delta）是否与原版一致
- Joker 效果的实际数值是否和 source oracle 对得上

所以 zero-warning 只能证明"引擎不会崩、走的是对的流程"，不能证明"算出来的分数和原版一样"。

#### 缺陷 3：lib.rs 已经 3830 行 — 单文件巨石

所有 engine 逻辑（状态机、计分、shop、consumable、retrigger、150 个 joker 效果、3 个阶段、测试）全挤在一个 `lib.rs` 里。可读性和可维护性很差。按照项目自己定的 800 行上限，这个文件已经超标 4.8 倍。

#### 缺陷 4：joker_on_played 阶段缺失

4 个 `joker_on_played` 类 Joker（Space Joker、DNA、To Do List、Midas Mask）完全没有独立激活阶段。它们应该在"计分牌确定后、per-card 计分前"触发，现在只是在 `apply_joker_effect` 里标了 supported 跳过了。

#### 缺陷 5：Card Enhancement 对计分的影响未实现

引擎虽然有 `enhancement: Option<String>` 字段，但计分时完全不读它。实际 Balatro 中：
- Bonus Card: +30 chips
- Mult Card: +4 mult
- Wild Card: 算作所有花色
- Glass Card: X2 mult (1/4 概率碎裂)
- Steel Card: X1.5 mult (held in hand)
- Stone Card: +50 chips (always scores)
- Gold Card: +$3 at end of round
- Lucky Card: 1/5 +20 mult, 1/15 +$20

这是一个系统性缺失，影响所有增强卡相关的 Joker 和 Tarot 效果正确性。

#### 缺陷 6：Edition（版本）对计分的影响未实现

Foil (+50 chips)、Holographic (+10 mult)、Polychrome (X1.5 mult)、Negative (+1 Joker slot) — 这些 Edition 效果对 Joker 和 playing card 都适用，目前完全缺失。

#### 缺陷 7：Boss Blind 特殊效果未实现

目前所有 Boss Blind 都当普通 Blind 处理。实际每个 Boss 有独特的负面效果：
- The Hook: 每手弃 2 张
- The Wall: X2 基础筹码
- The Wheel: 1/7 概率翻面
- The Arm: 降低打出牌型等级
- 等等约 25 种 Boss

#### 缺陷 8：Voucher 系统完全缺失

Voucher 是 Shop 里的永久升级（扩大手牌上限、Joker 槽位、折扣等），对游戏经济和决策空间影响很大。当前没有任何实现。

#### 缺陷 9：Booster Pack 完全缺失

Arcana Pack、Celestial Pack、Spectral Pack、Standard Pack、Buffoon Pack — 这些是 Shop 里的重要购买选项，打开后有选牌/选 Joker 的子决策。完全缺失。

#### 缺陷 10：真实客户端 trajectory 录制仍未打通

所有验证都是 engine 自对照（同 seed 两次跑一样）。还没有把真实 Balatro 客户端的 trajectory 拿来做对比验证。这是整个项目的"金标准"线，目前是 0 进度。

## 改进方向反思

### 1. 别再用 `supported = true` 自欺欺人

最大的教训是：**审计系统检查的颗粒度决定了"通过"的含金量**。当审计只检查 `supported` flag 时，把 flag 设成 true 就能通过 —— 这不是 fidelity，这是 checkbox driven development。

**改进**：B-01 是最高优先。必须在 audit 中加入"数值快照对比"层。哪怕一开始只对 chips/mult/money 三个字段做 source-oracle diff，也比现在的纯结构审计有意义得多。

### 2. 系统性缺失比逐个 Joker 重要

花了大量精力在 150 个 Joker match arm 上，但忽略了 Enhancement、Edition、Boss Effect、Voucher 这些**系统级机制**。一个正确实现了 Enhancement 计分的引擎比一个"150 个 Joker 都有分支但 Enhancement 不算分"的引擎有价值得多。

**改进**：下一轮优先做 B-03（Enhancement）和 B-04（Boss），再补 Joker 细节。

### 3. 单文件巨石必须拆

3830 行 lib.rs 是三个 parallel agent 合并的产物。每个 agent 单独工作时代码结构还行，合并后变成了意大利面。这个文件已经超过了人类或 agent 有效处理的上限。

**改进**：在做下一批功能之前先做 B-11（拆分）。否则后续每次改动都会更痛苦。

### 4. 真实客户端验真是零

这是整个项目最大的战略风险。所有"fidelity"验证都是 engine 自己跟自己对照。没有任何一条 trajectory 是从真实 Balatro 客户端录出来的。如果 engine 的基础假设就是错的（比如 hand type 判定逻辑、RNG 消耗顺序），自对照永远发现不了。

**改进**：B-12 应该尽早启动。哪怕先只录一条 5-step 的真实 trajectory 做 diff，也比 0 条强无穷倍。

### 5. Agent 并行合并的教训

这轮用了 3 个 worktree agent 并行改同一个文件，虽然最终合并成功了，但过程很脆弱。合并 agent 中途因配额耗尽失败，需要手动补救。

**改进**：先拆文件再并行。让不同 agent 改不同文件，避免合并地狱。或者用更清晰的 interface 边界让 agent 各自 own 一个模块。
